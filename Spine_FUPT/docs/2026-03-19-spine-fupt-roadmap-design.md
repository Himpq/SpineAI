## 1. 项目现状

| 维度 | 现状 |
|------|------|
| **后端** | Flask 单文件 (app.py ~3530行)，SQLite，19张表，~55个API，Session认证 |
| **前端** | Flutter App，37个Dart文件，15个页面，Riverpod + GoRouter + WebSocket |
| **AI** | 4种模型（关键点/分割/分类/边缘），远程推理服务已可用 |
| **测试** | ❌ 后端零测试，前端仅模板测试 |
| **部署** | 手动 `python app.py`，无容器化/CI |

## 3. Vibe Coding 工作循环

每个模块遵循统一的工作循环：

```
┌─────────────────────────────────────────────┐
│  1. AI 分析模块 → 写代码/修代码             │
│  2. AI 输出「真机测试步骤清单」              │
│  3. 你在手机上按步骤逐条验证                 │
│  4. 反馈结果（截图/描述）                    │
│  5. 有 bug → AI 修复 → 回到步骤 2           │
│     无 bug → ✅ 模块完成 → 下一个模块        │
└─────────────────────────────────────────────┘
```

**关键原则**：
- AI 主导代码编写，你主导验证
- 每个模块完成后构建新 APK
- 每次会话聚焦 1 个模块，不跨模块混做

## 4. 执行阶段

### 第1阶段：核心流程走通 + Bug 修复 (~3周, ~50h)

按 5 条核心业务线逐个攻克，每条线 = 若干模块。

#### 流程1：医生日常工作流 (~1周)

| 模块 | AI 任务 | 你的测试步骤 |
|------|---------|-------------|
| M1.1 登录/登出 | 审计登录逻辑、修复已知退出问题 | ① 输入错误密码→看报错 ② 正确登录→进入首页 ③ 退出→回到登录页 ④ 切后台再回来→仍在登录态 |

### 第2阶段：功能补全 + 体验打磨 (~3周, ~50h)

根据第1阶段实际发现的缺失来排优先级，预计的候选项：

| 优先级 | 功能 | 预估耗时 |
|--------|------|----------|
| P0 | 患者登记实时同步（二维码扫码→填信息→医生端实时看到） | 8-10h |
| P0 | App 内通知系统（新消息/新检查/问卷回收提醒） | 8-10h |
| P1 | 网络异常处理（断网提示、请求重试、友好错误信息） | 5-6h |
| P1 | 加载状态统一（骨架屏/Shimmer） | 4-5h |
| P2 | 消息/患者列表分页加载 | 4-5h |
| P2 | 后台恢复自动刷新 | 3-4h |
| P2 | 患者详情页增强（编辑信息、删除检查等） | 4-5h |

> 实际优先级在第1阶段完成后根据发现动态调整

### 第3阶段：测试覆盖 (~2周, ~40h)

| 层级 | 工具 | 目标数量 | AI 任务 |
|------|------|----------|---------|
| 后端 API 测试 | pytest + Flask test_client | ~30 用例 | AI 按模块写 pytest 测试 |
| Flutter 单元测试 | flutter_test + mockito | ~20 用例 | AI 为 Repository 层写 mock 测试 |
| Flutter Widget 测试 | flutter_test | ~10 用例 | AI 为关键页面写渲染+交互测试 |
| 集成冒烟测试 | integration_test | 1-2 条 | AI 写登录→患者列表→详情链路 |

**工作循环**：AI 写测试 → 你跑 `pytest` / `flutter test` → 反馈失败项 → AI 修 → 全绿 → 下一组

| 风险 | 应对 |
|------|------|
| 后端3500行单文件难修改 | 不拆重构，在现有结构上精准修补，用测试锁住 |
